## Phase 1 ‚Äî Frontend (biggest perceived-speed wins, do first)

### Step 1.1 ‚Äî Defer libphonenumber-js out of the initial/shared bundle (~120KB) ‚≠ê highest win
**Files:** `src/lib/utils/phone.ts`, `src/lib/validation/schemas.ts`, `src/components/forms/PhoneField.tsx`, `LoginForm.tsx`, `ContactForm.tsx`, `ProfileForm.tsx`, `src/components/cooperation/CooperationForm.tsx`.

- The killer is `phone.ts:27` ‚Äî `import examples from 'libphonenumber-js/examples.mobile.json'` ‚Äî a large JSON metadata blob pulled eagerly. Make phone parsing **lazy**: convert the top-level `import { ‚Ä¶ } from 'libphonenumber-js/min'` + `examples` import into `await import()` inside the functions that use them (validation on submit / on blur), not at module load.
- In `schemas.ts:2`, `isValidPhoneNumber` runs Zod-side. Split the phone refinement into a **lazily-loaded async validator** used only in the form's submit handler, keeping the base Zod schema synchronous and libphonenumber-free. For the common case (Iran `09XXXXXXXXX`), `phone.ts:83` already has a regex fast-path ‚Äî validate that **synchronously with a regex**, and only `await import('libphonenumber-js/min')` for the rare non-`09` / international `+` case.
- `PhoneField.tsx` (the `AsYouType` live formatter): gate the `AsYouType` import behind the same dynamic import; render the raw digits until it resolves. Login is the one place users type a phone ‚Äî a 1-frame delay to load the formatter is imperceptible vs. shipping 120KB to every anonymous homepage visitor.

**Expected:** removes ~120KB from the shared chunk that currently loads on `/` and `/prices` where no phone field is even visible.

### Step 1.2 ‚Äî Font subsetting for Persian (167KB ‚Üí ~40-60KB preload) ‚≠ê
**Files:** `public/fonts/Estedad.var.woff2` (121KB), `public/fonts/Vazirmatn.var.woff2` (46KB), `src/lib/theme/fonts.ts`.

- **Estedad is 121KB and is the display font** ‚Äî it likely still ships full Latin + all weights. Subset both `.var.woff2` to the **Persian/Arabic + digits + common punctuation** unicode ranges using `fonttools`/`glyphhanger` (`pyftsubset Estedad.var.woff2 --unicodes=U+0600-06FF,U+0750-077F,U+FB50-FDFF,U+FE70-FEFF,U+0020-007E,U+200C,U+2066-2069 --flavor=woff2 --output-file=Estedad.var.woff2`). Persian needs ZWNJ (U+200C) and the bidi marks ‚Äî include them.
- Add `unicodeRange` to the `localFont()` calls in `fonts.ts` matching the subset so the browser knows the coverage.
- Confirm Estedad is genuinely needed above the fold; if it's only headings, keep `preload: true` but on the subset; Inter already correctly has `preload: false`.

**Expected:** cuts the 167KB critical-path font preload by ~60-70% ‚Äî this is bytes on the LCP path over Iran latency, so it directly moves perceived first-paint.

### Step 1.3 ‚Äî Reduce AuthHydrator cost / kill the `/api/me` round-trip for anonymous users
**Files:** `src/lib/providers/AuthHydrator.tsx`, `src/lib/api` (auth client), optionally a lightweight cookie presence check.

- Today **every** visitor pays a `/api/me` fetch ‚Üí DB read. Gate it: only call `api.auth.me()` if a **session cookie is present**. The access cookie is httpOnly so JS can't read it directly ‚Äî add a **non-httpOnly, non-sensitive "has_session" marker cookie** (set/cleared alongside the real session cookies in `lib/auth/session.ts`) that `AuthHydrator` reads via `document.cookie`; skip the fetch entirely when absent. Anonymous visitors (the ~100% majority per the layout's own comment) then make **zero** auth requests.
- Also defer the fetch off the critical path: run it after first paint (it already is in `useEffect`, but ensure it doesn't block hydration of visible chrome ‚Äî it doesn't render anything, returns `null`, so this is fine; the win is skipping it entirely for anon).
- `useRequestsSync()` runs unconditionally too ‚Äî gate it behind the same "authenticated" signal.

### Step 1.4 ‚Äî LCP / image
**Files:** `next.config.mjs` (images block), homepage hero components under `src/app/page.tsx`.
- Images config is good (`avif`/`webp`). Ensure the homepage LCP element (hero/first price card) uses `next/image` with `priority` and explicit `sizes`; the logo is already preloaded (fine, it's small). Audit `/` for any non-`next/image` `<img>` in the hero.

### Step 1.5 ‚Äî Trim remaining chunks
- Keep `optimizePackageImports` and **add** `libphonenumber-js` is not needed once lazy. Consider adding `date-fns-jalali` (already there). Run `ANALYZE` build to confirm no other heavy lib leaked into shared (check `@tanstack/react-query`, `next-intl` client bundle from `LocaleProvider`).

## Phase 2 ‚Äî Redis: wire the existing (unused) cache abstraction

The abstraction in `redis.ts` is already clean ‚Äî **the work is calling it**, adding invalidation, and moving two stores onto it. No new client needed (ioredis present).

### Step 2.1 ‚Äî Read-through cache for hot catalog/price reads ‚≠ê (server offload)
**Files:** `src/lib/server/catalog.ts` (wrap `getCategories`, `getRows`, `getSubRows`, `findSku`), using `cacheGetJson`/`cacheSetJson` from `redis.ts`.

- Wrap the **live-mode** repo reads (not the mock branch). Pattern per function:
  ```
  const key = `cat:rows:${categorySlug}:${subSlug ?? '_'}`;
  const hit = await cacheGetJson<PriceRow[]>(key);
  if (hit) return hit;
  const rows = await repo.tableRows(categorySlug, subSlug);
  await cacheSetJson(key, rows, 120);   // TTL 120s
  return rows;
  ```
- **Keys & TTLs:**
  - `cat:categories` (the 7 categories, drives all chrome) ‚Äî TTL **300s**.
  - `cat:rows:<cat>:<sub>` (price tables) ‚Äî TTL **60‚Äì120s** (prices are the freshest data; keep short).
  - `cat:sku:<slug>` (`findSku`) ‚Äî TTL **120s**.
  - `cat:series:<slug>:<range>` (`priceSeries`, expensive history query) ‚Äî TTL **300s**.
- Note: ISR already caches the *rendered page* for 300s, so this Redis layer's value is (a) on-demand/dynamic reads (search, AI tools calling `tableRows`), (b) surviving ISR revalidation stampedes, and (c) the AI pipeline's tool calls. Don't double-cache what ISR already fully serves statically; prioritize the **AI tool path and search**, which are dynamic.

### Step 2.2 ‚Äî Cache invalidation on admin price edits ‚≠ê
**File:** `src/app/api/admin/pricing/route.ts` (`PUTImpl`, right where `safeRevalidatePath('/prices','layout')` already is).
- After a successful `savePrices`, in addition to the existing `revalidatePath`, call `cacheDel` for the affected keys. Since a bulk save can touch many SKUs across cats, simplest correct approach: maintain keys under a **version namespace** ‚Äî store `cat:ver` integer, include it in every cache key (`cat:rows:v<ver>:‚Ä¶`), and on any price write do `await getRedis()?.incr('cat:ver')`. One `INCR` atomically invalidates the whole catalog cache without `KEYS`/`SCAN` (which you must never run on the request path). Add a small `cacheVersion()` helper to `redis.ts`.
- Also invalidate on catalog CRUD routes: `admin/catalog/skus`, `subcategories`, `categories` (bump `cat:ver`), and `admin/market/billet`.

### Step 2.3 ‚Äî Move OTP store to Redis (latency + correctness)
**Files:** `src/lib/auth/store.pg.ts` (`otpCodes`, `otpRateLimits` handling), new `src/lib/auth/store.redis-otp.ts` or extend the store facade in `store.ts`.
- OTP codes are ephemeral (2‚Äì5 min TTL) and written/read on the hot login path; Postgres round-trips for a throwaway code are wasteful. Store `otp:<mobile>` ‚Üí `{code, attempts, expiresAt}` with Redis `SET ‚Ä¶ EX 120` and `INCR` for attempts, `otp:cooldown:<mobile>` for resend cooldown. **Keep the DB path as fallback** when `getRedis()` returns null (mirror the existing graceful-degradation pattern). TTLs: code **120s**, resend cooldown **60s**, attempt-lock window **300s** ‚Äî match the current constants in `lib/auth/service.ts`.

### Step 2.4 ‚Äî AI conversation session hot-path cache
**Files:** `src/lib/server/ai/conversation.ts`.
- The rolling **summary** and recent-turn fetch (`conversation.ts:169-183`) hit Postgres on every `/api/ai/chat` turn. Cache `ai:conv:<id>:summary` and `ai:conv:<id>:recent` in Redis (TTL **1800s**, invalidate on write). Keep Postgres as the durable source of truth (it must stay ‚Äî grounding/audit) but read-through Redis to cut per-turn DB latency during an active streaming conversation. **Do not** cache anything that feeds the grounding ledger ‚Äî only the summary/context, consistent with the file's existing "summary is context only" invariant.
- Rate-limit for AI chat already goes through Redis via `redisRateCheck` ‚Äî no change.

### Step 2.5 ‚Äî Local dev/production parity
- `redis.ts` already returns null gracefully without `REDIS_URL`, so all the above degrade to DB/direct reads in dev. Confirm `.env`/`.env.example` document `REDIS_PASSWORD` (compose requires it). Add a startup log line when Redis connects vs. falls back, for ops visibility.

## Phase 3 ‚Äî Server / Edge (Caddy)

**File:** `/opt/ahantime/Caddyfile`

### Step 3.1 ‚Äî HTTP/3
- Enable QUIC/HTTP/3 (big win over lossy Iran networks ‚Äî 0-RTT resumption, no head-of-line blocking). Add a global option block:
  ```
  {
    servers { protocols h1 h2 h3 }
  }
  ```
  (Caddy serves h3 on :443/udp automatically once enabled; ensure the host firewall opens **UDP 443** and compose publishes `443:443/udp` on the caddy service ‚Äî currently only tcp `443:443` is mapped in `docker-compose.yml`.)

### Step 3.2 ‚Äî Long-lived immutable caching for static assets
- Next emits content-hashed `/_next/static/*`. Add in the Caddyfile site block:
  ```
  @immutable path /_next/static/* /fonts/*
  header @immutable Cache-Control "public, max-age=31536000, immutable"
  ```
  This makes repeat visits (and second pages) skip re-downloading the 167KB‚Üí~50KB fonts and all JS chunks entirely. Next's `start` already sets this for `/_next/static`, but assert it at the edge so it survives regardless.
- Do **not** cache HTML at Caddy (ISR/`revalidate` is handled inside Next; edge-caching HTML would break auth freshness and the `has_session` logic).

### Step 3.3 ‚Äî Keep compression; verify zstd is negotiated
- `encode zstd gzip` is already present and correct (`zstd` preferred, `gzip` fallback). No change; just confirm Brotli isn't expected (zstd is better where supported).
