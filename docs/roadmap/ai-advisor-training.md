## The loop design (how "training" actually works here)

Since DeepSeek offers no fine-tuning, improvement = **curated retrieval injected into the cached context, gated by a growing eval suite**:

1. **Capture** thumbs up/down + reason on each answer в†’ `ai_feedback`.
2. **Review** in admin: inspect conversation + feedback, write a "golden" corrected answer в†’ `ai_corrections` (curated Q&A store).
3. **Inject** curated corrections as a *retrieval tool* (extend `searchGuides` pattern в†’ `searchKnowledge`) so future similar questions get the vetted answer as grounded context. Numbers in corrections flow through the ledger exactly like guide excerpts.
4. **Grow evals**: one-click "promote failing conversation в†’ eval scenario" from admin, appended to a data-driven regression suite.
5. **Prompt version + A/B**: store prompt variants keyed by version; assign per-conversation; measure via `aiUsage` + feedback rate per version.

### Increment 3: inject curated corrections as grounded retrieval (the actual "learning")

This is where answers improve **without touching the model**. Extend the proven `searchGuides` path.

1. **New tool `searchKnowledge`** вЂ” `aiTools.ts` AI_TOOLS (aiTools.ts:18): a retrieval tool that queries **both** `ai_corrections` and published guides, ranking curated corrections first. In `runTool` (aiTools.ts:287) add a `case 'searchKnowledge'` that calls `searchCorrections(q)` + `searchPublishedGuides(q)`, returns `{ results:[{source:'correction'|'guide', title, answer/excerpt, slug? }] }`. **Numbers in the returned strings flow into the ledger via `addFromJson`** (grounding.ts:44-53, already scans tool strings) вЂ” so a corrected answer's grounded numbers pass the validator, exactly like guide excerpts do today (aiTools.ts:212 comment).
   - Simplest: **replace `searchGuides`'s body** to also consult corrections (keep the tool name to preserve the cache prefix вЂ” see caching note), rather than adding a new tool that grows the tool list.

2. **Prompt rule** вЂ” `AI_SYSTEM_PROMPT` (aiTools.ts:403), rule 9: extend to "prefer a matching curated answer (Ш·ШЁЩ‚ ЩѕШ§ШіШ® ШЄШЈЫЊЫЊШЇШґШЇЩ‡Щ” ШўЩ‡Щ†вЂЊШЄШ§ЫЊЩ…) when returned." **Changing the prompt busts the DeepSeek cache prefix** вЂ” do this deliberately and version it (Increment 5), not casually.

3. **Retrieval only when useful** (cost): corrections injected **only via the tool call**, i.e. only when the model chooses to search вЂ” no unconditional context bloat. Cap results (3) and excerpt length (reuse `GUIDE_EXCERPT_CHARS=400`, aiTools.ts:213). This keeps token budgets bounded exactly as `searchGuides` already is.

4. **Test**: add an eval scenario in `evals.test.ts` (`SCENARIOS`, evals.test.ts:157) вЂ” script a `searchKnowledge` round returning a seeded correction, assert the golden answer text appears and `violationsCaught===0`.

**Shippable value after Inc 3:** curated fixes actually change future answers. The loop is closed.

### Increment 5: prompt version management + A/B

1. **Prompt store** вЂ” use existing `settings` table (system.ts:36, key/jsonb) key `ai.prompt.versions`: `{ activeVersion, variants: { [v]: { prompt, weight } } }`. Admin settings page edits it.
2. **Version selection** вЂ” `buildChatMessages` (conversation.ts:90) currently hardcodes `AI_SYSTEM_PROMPT` (conversation.ts:94). Change to accept a `promptVersion`/prompt string; the route picks a variant per conversation (deterministic hash of `conversationId` в†’ stable assignment).
   - **Caching constraint**: each distinct prompt string is its own cache prefix. A/B with 2 variants = 2 cache prefixes, both still cached across their own traffic. Keep variant count tiny (2) so cache hit-rate stays high. Store which version served each request.
3. **Attribution** вЂ” add `promptVersion` (and optionally `retrievalUsed`) columns to `aiUsage` (system.ts:80). Admin dashboard: feedback рџ‘Ќ/рџ‘Ћ rate + avg cost + `cacheHitTokens` ratio **per prompt version** вЂ” the A/B readout. All data already flows through the one `aiUsage` insert (route.ts:203).

## Optimization confirmations (task item 3)

- **Prompt cache prefix**: confirmed intact (conversation.ts:16-18,94). **Do not** prepend retrieval to the system prompt вЂ” inject via tool results (Inc 3) so the prefix stays byte-identical. Only Increment 5 deliberately introduces a small, versioned set of prefixes.
- **Token budgets**: `max_tokens:2000` (deepseek.ts:70), `MAX_TOOL_ROUNDS:4`, rolling summary at 12 msgs (conversation.ts:27). Corrections capped at 3Г—400 chars. No change needed.
- **Retrieval only when useful**: tool-gated (model decides), never unconditional.
- **Cost telemetry**: `aiUsage` already records prompt/completion/cache/violations per request (route.ts:203-215). Extend with `promptVersion` + `retrievalUsed` for A/B readout вЂ” no new telemetry pipeline.
